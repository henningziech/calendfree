# Internationalization (i18n)

## Context

Calendfree's entire frontend UI is hardcoded in German. The platform needs to support English (default) and German, configurable at two levels: per-company for public-facing pages and per-user for the dashboard/admin area. Backend error messages remain in English — only user-visible frontend strings are translated.

## Scope

### 1. Data Model

**User model** — add:
- `language` (String, default `"en"`) — controls dashboard/admin UI language

**Company model** — add:
- `language` (String, default `"en"`) — controls public page language (booking, routing, confirmation, cancel/reschedule)

Both fields accept `"en"` or `"de"`. Stored in the database, delivered to the frontend via existing session/API responses.

### 2. i18n Infrastructure

**Libraries:** `react-i18next` + `i18next`

**File structure:**
```
frontend/src/i18n/
├── index.ts              # i18next init + config
├── locales/
│   ├── en/
│   │   ├── common.json   # Shared: buttons, labels, errors, loading states
│   │   ├── booking.json  # Public booking pages
│   │   ├── dashboard.json # User dashboard
│   │   ├── admin.json    # Admin panel
│   │   └── routing.json  # Routing forms (public + admin)
│   └── de/
│       ├── common.json
│       ├── booking.json
│       ├── dashboard.json
│       ├── admin.json
│       └── routing.json
```

**Namespaces:** 5 namespaces split by feature area. `common` contains shared strings used across multiple areas (Save, Cancel, Loading, Error, Delete, etc.).

**i18next config:**
- Default language: `"en"`
- Fallback language: `"en"`
- Namespaces loaded eagerly (no lazy loading for V1 — total string volume is small)
- No browser language detection — language comes from DB settings

### 3. Language Selection Logic

**LanguageProvider** wrapper component sits inside `I18nextProvider` and determines the active language by route context:

1. **Dashboard/Admin pages** (routes starting with `/dashboard` or `/admin`): Read `user.language` from auth context (session data). Call `i18next.changeLanguage(user.language)`.
2. **Public pages** (all other routes — booking, routing, confirmation, cancel, reschedule): Read `company.language` from the company info API response (`GET /api/booking/:companySlug/info`). Call `i18next.changeLanguage(company.language)`.
3. **Fallback:** `"en"` if no language is set.

The provider uses route matching (via `useLocation`) to determine which context applies. It wraps the app inside `BrowserRouter` and reacts to language changes from the respective data source. When an authenticated user visits a public page (e.g., previewing their own booking page), the company language takes precedence since the page is public-facing.

### 4. Frontend String Extraction

All ~44 components with ~300-500 hardcoded German strings must be converted to use `useTranslation()`.

**Pattern — inline text:**
```tsx
// Before
<button>Speichern</button>

// After
const { t } = useTranslation('common');
<button>{t('save')}</button>
```

**Pattern — component prop defaults:**
```tsx
// Before
export function LoadingSpinner({ text = 'Laden...' })

// After
export function LoadingSpinner({ text }: { text?: string }) {
  const { t } = useTranslation('common');
  return <span>{text ?? t('loading')}</span>;
}
```

**Pattern — data structures with labels:**
```tsx
// Before
const MAIN_NAV = [{ to: '/dashboard', label: 'Termine' }];

// After — use t() at render time, not at module level
function Sidebar() {
  const { t } = useTranslation('dashboard');
  const nav = [{ to: '/dashboard', label: t('nav.bookings') }];
}
```

**Date formatting:** Replace hardcoded `import { de } from 'date-fns/locale'` with dynamic locale selection based on active language. Create a utility that returns the correct `date-fns` locale (`de` or `enUS`).

**Translation key conventions:**
- Dot-notation for nesting: `nav.bookings`, `form.title`, `status.active`
- camelCase for multi-word keys: `noResults`, `confirmDelete`
- Interpolation: `t('greeting', { name })` → `"Hello, {{name}}"`
- Pluralization: `t('bookingCount', { count: n })` → `"{{count}} booking"` / `"{{count}} bookings"`

**Hardcoded German defaults in Zod schemas** (e.g., `CreateRoutingFormSchema.question` defaults to `"Wofür interessieren Sie sich?"`): These are backend-level defaults and remain as-is. They serve as German defaults for German-speaking companies. The frontend displays the actual stored value, not a translated default.

### 5. Settings UI

**User language** — Account Settings page (`/dashboard/settings`):
- Dropdown labeled "Language" (en) / "Sprache" (de)
- Options: "English", "Deutsch"
- Positioned next to the existing Timezone dropdown
- Saves via new `PATCH /api/me/language` endpoint (same pattern as existing `PATCH /api/me/timezone`) → `User.language`

**Company language** — Company Detail Page (`/admin/companies/:companyId`):
- Dropdown labeled "Public Page Language" (en) / "Sprache der öffentlichen Seiten" (de)
- Only visible/editable by COMPANY_ADMIN and ORG_ADMIN
- Saves via existing company PATCH endpoint → `Company.language`

### 6. API Changes

**Session data** (`GET /api/auth/me` or session object):
- Include `user.language` in the response (already part of user object once field is added)

**Company info** (`GET /api/booking/:companySlug/info`):
- Include `company.language` in the response

**User language** (`PATCH /api/me/language`):
- New endpoint, same pattern as `PATCH /api/me/timezone`
- Accept `{ language: "en" | "de" }`
- Updates `User.language` for the authenticated user

**Company PATCH** (`PATCH /api/admin/companies/:id`):
- Accept `language` field (`"en"` or `"de"`)

No backend i18n — error messages stay English.

**Out of scope:** Email notifications (booking confirmations, cancellations). These remain in their current language and may be addressed in a future iteration.

## Verification

1. **Default English:** Fresh install shows all UI in English
2. **German company:** Setting company language to "de" shows all public pages (booking, routing, confirmation, cancel, reschedule) in German
3. **German user:** Setting user language to "de" shows dashboard and admin in German
4. **Mixed:** Company=de + User=en → public pages German, dashboard English
5. **Date formatting:** Dates follow the active language (e.g., "Monday, March 17" vs "Montag, 17. März")
6. **Settings:** User can change language in Account Settings, Company language changeable by COMPANY_ADMIN/ORG_ADMIN
7. **Persistence:** Language choice survives logout/login (stored in DB)
8. **No missing strings:** All visible UI text is translated — no German strings appear when language is English
